经过无数的VFP坑之后……只能全部用androideabi吗……Emmmm

默认情况下，ARM 工具链生成的机器代码应与官方 Android armeabi ABI 兼容。
与此ABI兼容的设置有 -mfloat-abi=softfp 与 -mfloat-abi=soft，前者使用硬浮点运算，后者使用软浮点运算。
对于Cortex-A7内核，使用硬浮点，并利用NEON处理器，编译设置应该为：
-march=armv7-a -mfloat-abi=softfp -mfpu=neon-vfpv4
链接器应设置：
-march=armv7-a -Wl,--fix-cortex-a8
这样才能链接到定制的库。


一向信不过跑虚拟机的东西，如果效率浪费在这里就很坑了。

Linux环境优化实验

Android环境本身就复杂得要死，初学一看能玩对就有鬼了。还优化，玩鬼呢。
先到Linux层做实验。

优化思路

这其实是一个边写边剧透的实验报告

  1. 芯片相关的编译参数必须指定对了。库编译时也要注意扩展参数/宏。
    -marm -mfpu=neon-vfpv4 -march=armv7-a -mfloat-abi=[softfp|hard]
    无论是编译库，还是编译App，都必须指定这些参数。它们是全局的。
    一句话的事。必须在最终传递给gcc/clang的参数中确认。
    使用NEON后，人脸框定时间从2500ms降到850ms。
  2. 引入BLAS库。因为作者是这么说的。
    常用的是OpenBLAS库，经过优化，并且可以绕过Fortran编译。
    OpenBLAS工作在双线程，引入后，特征计算时间从3000ms降到400ms。
  3. 引入TCMalloc库。
    在OpenBLAS中可以看到大量的临时new/delete操作，dlib中存在内存管理类，具体还没看。引入优化的malloc操作可以节约时间。
    引入后，在符号表中确认到引用，但只有特征计算时间降低了约50ms。
    有待进一步尝试，并分析dlib基础代码的调用。
  4. 分析dlib识别人脸的代码，在合适的地方进行多线程优化，并考虑优化线程的调度。
  5. 考虑对不适合多线程加速的运算进行变形，改写为NEON或者BLAS调用。

对于NEON的设置，实际代码是以__ARM_NEON宏来开启DLIB_HAVE_NEON标志。但编译DLib的时候，判定的宏是__ARM_NEON__，这个宏应当由编译器设定，并且CMake会给出提示消息：
__ARM_NEON__ defined.
没有的话，说明编译器没自动设定，或者测试不通过/不支持。
正常情况下编译器会自动设定这两个宏。

因为CMake谜一样的搜索规则，注意Library Mode要设置为BOTH，否则找不到OpenBLAS。

设定以下项目：

  • CMAKE_BUILD_TYPE = Release
  • CMAKE_CXX_FLAGS = -march=armv7-a -mfloat-abi=hard -mfpu=neon-vfpv4 -fPIC
  • CMAKE_C_FLAGS = -march=armv7-a -mfloat-abi=hard -mfpu=neon-vfpv4 -fPIC
  • CMAKE_INSTALL_PREFIX = <自己设定安装目录>

确保Dlib能寻找到OpenBLAS库，以下标志应该被自动设置为：

  • DLIB_USE_BLAS = ON
  • cblas_lib = <OpenBLAS库位置>

关闭不需要、可能产生问题的选项：

  • DLIB_JPEG_SUPPORT = OFF

暂时动不了。

get_frontal_face_detector

恕我直言，这载入数据的写法就是傻【哔】。
函数总共耗时2.5s，其中有2.4s都在往istringstream里写数据。
做个独立文件会死？

在get_serialized_frontal_faces()后边设个卡，把解码数据写文件里，以后就直接读这个文件了。

做出独立文件后，载入时间降为30ms。

deserialize

deserialize(shape_predictor&, std::istream&)我基本快没话说了。
全体deserialize我都快没话说了。
居然直接用std::istream读取……
干得漂亮，我也不知道怎么优化。

之前没有优化std::istream的经验，因为根本就不敢用这货，网上给出的答案基本就是设置缓冲大小，或者干脆”istream完全不慢啊“。
实际测试，缓冲大小不影响反序列化速度。虽然我知道开一个大一点的缓冲是可以提速的，而且可以提很多。
更多的优化都注重于std::cin与std::cout的标准流的优化。
也有建议说别用istream，直接上fscanf或者mmap，然而这上百个deserialize我不可能全改了。

要不继承一个istream然后魔改？拉倒吧STL也敢改？
改也不知道往哪改啊……

可是deserialize一个反序列化的过程，只能按顺序一步步递归展开，而且没有数据标注一个数据结构的完整大小，所以并行是不可能的。

理了一下，其实所有的反序列化操作最终都归结于整数的反序列化，然而这个流程挑不出毛病。

玩蛋。

shape_predictor的反序列化要10s，我把循环合并整数的操作改成64位整数存取之后，甚至升到了13s。
face_recognition_model_v1::set_model同理，不过才600ms就拉倒了吧。
看不透。不好意思打扰了。
