硬件环境说明

MCU是全志A33。外设有什么我不管。
芯片数据手册说的很清楚：

2.1. CPU Architecture
The A33 platform is based on quad-core Cortex-A7 CPU architecture.

  • ARMv7 ISA standard instruction set plus Thumb-2 and Jazeller RCT
  • NEON with SIMD and VFPv4 support
    …

2.2. GPU

  • Mali400MP2 GPU
  • Support OpenGL ES 2.0 / OpenVG 1.1 standard

所以针对芯片的优化方向很明确：

  • 多线程绑核，用上4核
  • 编译生成ARM指令，在32位芯片上ARM指令比Thumb指令快/简短
  • 启用VFP(v4)指令
  • 启用NEON指令
  • GPU就别想了，只开放了OpenGL接口，OpenGLES的通用计算能玩死人

Android编译环境搭建

讲真，我也不知道这个编译环境到底是什么的编译环境。

Android下面跑了一层超级简化的Linux。
按正常套路来想，Linux直接在芯片上跑，Cortex-A7什么的直接套arm_eabi_hf就行了；
Linux跟Android对接，虽然不知道Android上面跑了一层虚拟机是怎么对接的，但反正就是定成android_eabi了。
经过无数的VFP坑之后……只能全部用androideabi吗……Emmmm

默认情况下，ARM 工具链生成的机器代码应与官方 Android armeabi ABI 兼容。
与此ABI兼容的设置有 -mfloat-abi=softfp 与 -mfloat-abi=soft，前者使用硬浮点运算，后者使用软浮点运算。
对于Cortex-A7内核，使用硬浮点，并利用NEON处理器，编译设置应该为：
-march=armv7-a -mfloat-abi=softfp -mfpu=neon-vfpv4
链接器应设置：
-march=armv7-a -Wl,--fix-cortex-a8
这样才能链接到定制的库。


一向信不过跑虚拟机的东西，如果效率浪费在这里就很坑了。

编译Linux的东西就到Linux里玩，到Windows上可能又缺一堆-dev库，玩蛋。

Linux环境优化实验

Android环境本身就复杂得要死，初学一看能玩对就有鬼了。还优化，玩鬼呢。
先到Linux层做实验。

优化思路

这其实是一个边写边剧透的实验报告

  1. 芯片相关的编译参数必须指定对了。库编译时也要注意扩展参数/宏。
    -marm -mfpu=neon-vfpv4 -march=armv7-a -mfloat-abi=[softfp|hard]
    一句话的事。必须在最终传递给gcc/clang的参数中确认。
  2. 引入BLAS库。因为作者是这么说的。
    常用的是OpenBLAS库，经过优化，并且可以绕过Fortran编译。
    OpenBLAS工作在双线程，引入后，特征计算时间从3000ms降到400ms。
  3. 引入TCMalloc库。
    在OpenBLAS中可以看到大量的临时new/delete操作，dlib中存在内存管理类，具体还没看。引入优化的malloc操作可以节约时间。
    引入后，在符号表中确认到引用，但只有特征计算时间降低了约50ms。
    有待进一步尝试，并分析dlib代码的调用。
  4. 分析dlib识别人脸的代码，检查是否有重复的计算/内存操作的大量调用。
    如果有，检查上面两个库的引入；如果没有，考虑多线程优化，或特定优化。

CMake

然后是CMake配置。
甚至不需要toolchain文件……直接Specify options for cross-compiling。

  • Operating System = Linux
  • Processor = arm
  • C Compiler = <Toolchain>/bin/arm-linux-gcc
  • C++ Compiler = <Toolchain>/bin/arm-linux-g++
  • Target Root = <Toolchain>/arm-cortexa9-linux-gnueabihf/sys-root
  • Program Mode = BOTH
  • Library Mode 看情况，一般ONLY
  • Include Mode = ONLY

该配的CMake都配好了。

Dlib编译

拉进CMake之前，先要改搜索OpenBLAS的规则。
打开Dlib源码目录下dlib/cmake_utils/find_blas.cmake，找到extra_paths的定义：

列表里加上刚刚编译安装的OpenBLAS库地址。

然后按上面的设定，用CMake配置Dlib。
因为CMake谜一样的搜索规则，注意Library Mode要设置为BOTH，否则找不到OpenBLAS。

设定以下项目：

  • CMAKE_BUILD_TYPE = Release
  • CMAKE_CXX_FLAGS = -march=armv7-a -mfloat-abi=hard -mfpu=neon-vfpv4 -fPIC
  • CMAKE_C_FLAGS = -march=armv7-a -mfloat-abi=hard -mfpu=neon-vfpv4 -fPIC
  • CMAKE_INSTALL_PREFIX = <自己设定安装目录>

确保Dlib能寻找到OpenBLAS库，以下标志应该被自动设置为：

  • DLIB_USE_BLAS = ON
  • cblas_lib = <OpenBLAS库位置>

关闭不需要、可能产生问题的选项：

  • DLIB_JPEG_SUPPORT = OFF

作为函数库，不需要明确外部调用，因此不需要链接tcmalloc。

最后生成Makefile，直接make -j4 install完事。
